# Clarify: 연구 기반 프롬프트 명확화

## 핵심 원칙

```
"모든 모호성에 질문하지 않는다. 추론 → 가정 명시 → 필요시 질문."
```

**순서**:
1. 먼저 컨텍스트에서 추론 시도
2. 추론 가능하면 가정을 명시하고 진행
3. 결과가 크게 달라지는 핵심 모호성만 질문 (필요한 만큼 N개)

**설계 철학**:
1. **추론 우선** - AI는 바보가 아니다. 먼저 추론하고, 가정을 명시하고, 필요시 질문
2. **Taxonomy 기반 분류** - 감으로 판단하지 않고, 체계적으로 분류
3. **Information Gain 우선** - 결과가 크게 달라지는 핵심 모호성에만 질문
4. **필요한 만큼만** - 1개로 충분하면 1개, 여러 개 필요하면 N개 (권장: 1~3개)

## 1. 프롬프트 3요소 분석 (P-S-O Framework)

모든 프롬프트는 세 가지 핵심 요소로 구성된다:

| 요소 | 질문 | 모호할 때의 문제 |
|------|------|-----------------|
| **Purpose (목적)** | 왜 하는가? | 방향을 잘못 잡음 |
| **Subject (주제)** | 무엇에 대한 것인가? | 엉뚱한 대상을 다룸 |
| **Objective (목표)** | 어떤 결과를 원하는가? | 기대와 다른 산출물 |

### 분석 예시

```
입력: "이 코드 좀 정리해줘"

분석:
- Purpose: 불명확 (왜 정리? 가독성? 성능? 유지보수?)
- Subject: 부분 명확 ("이 코드" = 현재 파일, 추론 가능)
- Objective: 불명확 (어떤 상태가 "정리된" 것?)
```

## 2. 모호성 분류 체계 (CLAMBER 기반)

### 2.1 Linguistic Ambiguity (언어적 모호성)

단어나 문장 자체가 여러 해석을 허용

| 하위 유형 | 설명 | 예시 |
|----------|------|------|
| **Lexical** | 단어의 다의성 | "최적화" (시간? 공간? 가독성?) |
| **Semantic** | 문장 의미 모호 | "깔끔하게" (코드 스타일? UI?) |
| **Referential** | 대상 불명확 | "그거", "아까 그것" |

### 2.2 Aleatoric Output (정보 부족)

필요한 정보가 누락됨

| 누락 유형 | 질문 | 예시 |
|----------|------|------|
| **What** | 무엇을? | "추가해줘" (뭘?) |
| **Whom/Which** | 어떤 것을? | "파일 수정해줘" (어떤 파일?) |
| **How** | 어떻게? | "개선해줘" (어떤 방식?) |
| **How much** | 얼마나? | "몇 개 만들어줘" |

### 2.3 Epistemic Misalignment (지식 불일치)

맥락이나 배경 지식의 차이

| 하위 유형 | 설명 | 예시 |
|----------|------|------|
| **Assumption Gap** | 전제 불명확 | "우리 프로젝트 스타일로" |
| **Contradiction** | 요구사항 충돌 | "빠르고 안정적으로" |
| **Domain Specific** | 전문 용어 | "SOLID하게 리팩토링" |

## 3. Information Gain 기반 질문 선택

### 3.1 질문 효용성 평가

모호성이 여러 개일 때, **가장 해소 효과가 큰** 것을 선택:

```
Information Gain = 해소 시 결과 차이 × 실행 영향도
```

| 모호성 | 해소 시 차이 | 영향도 | IG Score |
|--------|------------|--------|----------|
| 목적(Purpose) | 방향 전환 | 높음 | ⭐⭐⭐ |
| 목표(Objective) | 결과물 변경 | 높음 | ⭐⭐⭐ |
| 주제(Subject) | 대상 변경 | 중간 | ⭐⭐ |
| 수량/범위 | 양적 차이 | 낮음 | ⭐ |

### 3.2 질문 우선순위

```
1순위: Purpose 또는 Objective가 완전히 불명확
2순위: Subject가 불명확 + 여러 후보 존재
3순위: 세부사항 (수량, 스타일 등)
```

**중요**: 3순위는 질문하지 않고 합리적 기본값으로 진행

## 4. 질문 조건 (When to Ask)

### 4.1 질문해야 하는 경우 ✅

```yaml
조건:
  - P-S-O 중 2개 이상 불명확
  - 또는 상반된 해석이 가능하고 결과가 크게 다름
  - 또는 파괴적 작업 + 범위 불명확
```

### 4.2 질문하지 않는 경우 ❌

```yaml
조건:
  - 컨텍스트에서 추론 가능
  - 합리적 기본값 존재
  - 되돌리기 쉬운 변경
  - 세부사항만 불명확 (3순위)
```

## 5. AskUserQuestion 사용법

### 5.1 필수 규칙

1. **분석 과정 출력 금지** - 사용자에게 taxonomy, score 노출하지 않음
2. **추론 우선** - 질문 전에 반드시 컨텍스트에서 추론 시도
3. **가정 명시** - 추론했다면 가정을 명시하고 진행 ("~라고 가정하고 진행합니다")
4. **필요한 만큼 질문** - 1개로 충분하면 1개, 여러 개 필요하면 N개 (권장: 1~3개)
5. **2-3개 선택지** - 너무 많으면 부담
6. **마지막 옵션** - 항상 "기존대로 진행" 또는 "제가 판단"

### 5.2 질문 구성

```yaml
question: "[P-S-O 중 불명확한 요소]에 대해 확인이 필요합니다"
header: "[간결한 제목]"
options:
  - label: "[구체적 해석 A]"
    description: "[이 선택 시 결과]"
  - label: "[구체적 해석 B]"
    description: "[이 선택 시 결과]"
  - label: "제가 판단해서 진행"
    description: "가장 일반적인 방식으로 진행"
```

## 6. 모호성 유형별 질문 템플릿

### 6.1 Purpose (목적) 불명확

```
입력: "이 함수 고쳐줘"
분석: 왜 고치는지 불명확 (버그? 성능? 가독성?)

question: "어떤 문제를 해결하려고 하시나요?"
options:
  - label: "버그/오류 수정"
    description: "잘못된 동작을 고칩니다"
  - label: "성능 개선"
    description: "더 빠르게 동작하도록 합니다"
  - label: "제가 판단해서 진행"
    description: "코드를 분석해서 가장 필요한 개선을 합니다"
```

### 6.2 Subject (주제) 불명확

```
입력: "테스트 추가해줘"
분석: 어떤 대상의 테스트인지 불명확

question: "어떤 부분의 테스트를 추가할까요?"
options:
  - label: "현재 열린 파일"
    description: "auth.ts의 함수들을 테스트"
  - label: "최근 변경된 파일들"
    description: "마지막 커밋의 변경사항 테스트"
  - label: "제가 판단해서 진행"
    description: "테스트 커버리지가 낮은 부분부터"
```

### 6.3 Objective (목표) 불명확

```
입력: "이 컴포넌트 개선해줘"
분석: 어떤 상태가 "개선된" 것인지 불명확

question: "어떤 방향으로 개선할까요?"
options:
  - label: "코드 품질"
    description: "가독성, 구조, 재사용성 개선"
  - label: "기능 확장"
    description: "새로운 props, 상태 추가"
  - label: "제가 판단해서 진행"
    description: "현재 코드의 문제점을 분석해서 개선"
```

### 6.4 상반된 해석 가능

```
입력: "로그인 구현해줘"
분석: 여러 구현 방식 가능, 결과가 크게 다름

question: "어떤 방식의 로그인을 원하시나요?"
options:
  - label: "이메일/비밀번호"
    description: "전통적인 폼 기반 로그인"
  - label: "OAuth/소셜 로그인"
    description: "Google, GitHub 등 연동"
  - label: "제가 판단해서 진행"
    description: "프로젝트 설정을 보고 적합한 방식 선택"
```

## 7. 컨텍스트 활용 (질문 감소)

질문 전에 **반드시** 다음 컨텍스트 확인:

### 7.1 세션 컨텍스트
- 현재 열린 파일
- 최근 편집 위치
- 이전 대화 내용
- 최근 에러 로그

### 7.2 프로젝트 컨텍스트
- CLAUDE.md, README.md
- 기존 코드 패턴
- 프로젝트 구조
- 사용 중인 기술 스택

### 7.3 추론 규칙

| 표현 | 추론 대상 |
|------|----------|
| "이거", "여기" | 현재 열린 파일/커서 위치 |
| "아까 그거" | 이전 대화에서 언급된 항목 |
| "에러 고쳐줘" | 가장 최근 에러 |
| "마저 해줘" | 이전 작업의 연속 |
| 동사만 있음 | 현재 컨텍스트에서 가장 관련 있는 대상 |

## 9. 예시 시나리오

### 시나리오 1: 추론으로 해결

```
[상황]
- 열린 파일: src/components/Button.tsx
- 최근 에러: TypeScript 타입 에러 (line 42)

[입력]
"여기 에러 고쳐줘"

[분석]
- Purpose: 명확 (에러 수정)
- Subject: 추론 가능 (현재 파일 + 에러 위치)
- Objective: 명확 (에러 해소)

[결과]
→ 질문 없이 바로 진행
→ "Button.tsx line 42의 타입 에러를 수정합니다."
```

### 시나리오 2: 질문 필요

```
[상황]
- 새 프로젝트
- 인증 관련 설정 없음

[입력]
"로그인 기능 만들어줘"

[분석]
- Purpose: 부분 명확 (인증)
- Subject: 명확 (로그인)
- Objective: 불명확 (어떤 방식? 어떤 범위?)

[Information Gain]
- "어떤 인증 방식?" → 높음 (결과가 완전히 다름)
- "UI도 포함?" → 낮음 (나중에 추가 가능)

[결과]
→ 인증 방식에 대해서만 질문

question: "어떤 로그인 방식을 원하시나요?"
options:
  - label: "이메일/비밀번호"
  - label: "OAuth (Google, GitHub 등)"  
  - label: "제가 판단해서 진행"
```

### 시나리오 3: 기본값으로 진행

```
[입력]
"버튼 컴포넌트 몇 개 만들어줘"

[분석]
- Purpose: 명확 (UI 구성)
- Subject: 명확 (버튼)
- Objective: 부분 불명확 (수량만)

[Information Gain]
- 수량 → 낮음 (3순위, 양적 차이만)

[결과]
→ 질문하지 않고 합리적 기본값(3개)으로 진행
→ "Primary, Secondary, Outline 3가지 버튼을 만들겠습니다."
```

## 10. 안티패턴 (하지 말 것)

### ❌ 분석 과정 노출
```
나쁜 예: "Linguistic Ambiguity가 감지되었습니다. 
         Lexical 유형으로 '최적화'라는 단어가..."
```

### ❌ 다중 질문
```
나쁜 예: "1) 어떤 파일? 2) 어떤 방식? 3) 테스트 포함?"
```

### ❌ 3순위 모호성에 질문
```
나쁜 예: "몇 개 만들까요?" (기본값으로 진행해야 함)
```

### ❌ 추론 가능한데 질문
```
나쁜 예: (auth.ts 열린 상태에서) "어떤 파일을 수정할까요?"
```

## 11. 평가 지표

| 지표 | 목표 | 설명 |
|------|------|------|
| 질문 발생률 | < 30% | 불필요한 질문 최소화 |
| 질문 유효성 | > 90% | 질문이 실제로 결과에 영향 |
| 1회 질문 해소율 | > 85% | 한 번 질문으로 충분 |
| 컨텍스트 활용률 | > 70% | 추론으로 해결한 비율 |
