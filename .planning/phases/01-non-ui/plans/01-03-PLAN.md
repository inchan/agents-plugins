  <action>
    v0.2의 `commands/ticket.md`를 기반으로 `commands/ticket-workflow.md`를 새로 생성한다.

    1. YAML frontmatter:
       ```yaml
       ---
       description: "Process a bug ticket through evidence collection, classification, exploration, planning, implementation, and verification phases"
       argument-hint: "[--type ui|non-ui] <ticket description or URL>"
       allowed-tools: Bash, Read, Edit, Write, Glob, Grep, Agent, WebFetch, WebSearch
       ---
       ```

    2. 커맨드명: `# /ticket-workflow -- Bug Ticket Workflow Entry Point`

    3. v0.2 ticket.md의 구조를 그대로 가져오되 다음을 변경:
       - `/ticket` -> `/ticket-workflow` 모든 언급
       - `lib/schemas/classification.md` 참조 -> `references/classification.md`로 변경
       - Reference Map의 모든 경로가 `skills/ticket-workflow/references/` 기준으로 유효한지 확인

    4. INIT 섹션에 시작 배너 유지:
       ```
       [INFO ] [WORKFLOW] +0:00 -- Initializing 6 workflow phases...
       ```

    5. Phase Pipeline 섹션에서 6단계 순차 실행 명시:
       - Phase 1 EVIDENCE: 2x evidence-searcher 병렬 디스패치
       - Phase 2 CLASSIFY: classification.md 참조
       - Phase 3 EXPLORE: 2-3x bug-tracer 병렬 디스패치
       - Phase 4 PLAN: planning.md 참조
       - Phase 5 IMPLEMENT: implementation.md 참조
       - Phase 6 VERIFY: verification.md 참조

    6. Retry Control Flow 섹션에 3단계 전략 변경을 명시:
       ```
       Attempt 1 (full): 원래 계획 그대로 전체 구현 + 검증
         FAIL -> [WARN] [VERIFY] -- Attempt 1/3 failed: {reason}. Retry strategy: targeted
       Attempt 2 (targeted): 실패 원인 분석 후 해당 부분만 수정. 정상 부분 유지
         FAIL -> [WARN] [VERIFY] -- Attempt 2/3 failed: {reason}. Retry strategy: relaxed
       Attempt 3 (relaxed): 접근 방식 변경. 다른 파일/함수 수정 또는 우회 전략
         FAIL -> [ERROR] [VERIFY] -- All attempts exhausted -> FAILED
       ```

    7. "Retry N/3 -- 전략: targeted/relaxed" 형태의 한글+영어 혼용 로그 패턴 명시 (성공 기준 3항 충족)

    8. Phase 1-4 불변 규칙 명시: "Phases 1-4 outputs remain immutable during retries. Only Phases 5-6 re-execute."

         * Attempt 1 (full): 원래 계획 그대로. verification_strategy 변경 없음
         * Attempt 2 (targeted): failure_reason에서 실패 지점 식별. 해당 파일/함수만 수정. 이전 정상 변경은 유지
         * Attempt 3 (relaxed): 접근 방식 자체를 변경. 다른 파일 수정, 우회 전략, 또는 대안 알고리즘 적용
       - 각 재시도 전환 시 로그 포맷 명시:
         `[INFO ] [VERIFY] +<time> -- Retry N/3 -- 전략: <strategy>`
       - retry_context 구조에 `strategy: "full"|"targeted"|"relaxed"` 필드 추가
       - Rule: "Phases 1-4 outputs remain immutable during retries" 강조

    2. verification.md 업데이트:
       - Non-UI 검증 절차를 더 상세히:
         * 테스트 실행 (기존 + 신규 테스트)
         * assertion 결과 분석
         * 로그 분석
         * 사이드 이펙트 확인 (다른 테스트가 깨지지 않았는지)
       - retry_context 수신 시 전략별 행동 분기:
         * targeted: 실패 테스트만 집중 분석, 해당 코드만 수정
         * relaxed: 전체 접근법 재고, 다른 수정 경로 탐색
       - UI 검증 부분은 "Phase 2에서 browser-reproducer 통합 예정" 주석으로 유지
  </action>
